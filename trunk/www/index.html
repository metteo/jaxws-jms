Welcome to the JMS Web service transport project, an example Web service transport component for the JAX-WS reference implementation 2.0.1 that utilizes JMS.

Introduction

This project is designed to show by example how to implement plugable transports into the latest (rearchitectured) JAX-WS code, which supports client and service-based transports as plugable components.

The example chosen in this case is JMS. SOAP messages produced and consumed by the JAX-WS runtime are transmitted over a JMS Queues.

How to run the sample

A sample Web service is provided (see here) that is configured to use the JMS transport.

To run the sample it is required that the FCS build of glassfish is used.

The sample can be built and run in the following steps:

  • Start the Glassfish application server.

  • Create a JMS factory named 'jaxwsfactory' and a JMS queue named 'jaxwsqueue'.
    An example on how to create factories and queues using the Web admistration application may be found here.

  • Build the JMS transport component.
    Type 'ant' from the main project directory.

  • Build the sample Web service War file.
    Type 'ant server' from the samples/fromwsdl directory.

  • Deploy and run the Web service as a standalone process.
    Type 'ant run-service' from the samples/fromwsdl directory.

  • Build and run the client.
    Type 'ant client run-client' from the samples/fromwsdl directory.

How the sample works with the JMS transport

To "plug-in" a transport requires no modification to the following: client or service endpoint implementation code; and deployment descriptors. Everything is specified from the endpoint address URL, the scheme of which determines what transport is used in JAX-WS.

The JMS transport uses a configured queue for the request and a temporary queue for the reply to suport the request response message exchange pattern.

The WSDL (AddNumbers.wsdl) used by the sample contains the following soap:address for the AddNumbersPort of the AddNumbersService: 

<soap:address location="x-jms://localhost:3700/jaxwsfactory/jaxwsqueue?path=fromjava/addnumbers" />

The location attribute value is a URL whose scheme is "x-jms". This informs the JAX-WS client runtime that the JMS client transport should be used. The sample client includes in its class path a file in the META-INF/services directory called com.sun.xml.ws.api.pipe.TransportPipeFactory. This file contains the class of the JMS transport component, com.sun.xml.ws.transport.jms.client.JMSTransportPipeFactory. The inclusion of this file registers the JMS transport factory with the JAX-WS client runtime. The factory knows about the 'x-jms' URL scheme and is responsible for instantiating a JMS transport pipe.

The URL for the JMS transport has the following structure:

x-jms://<host>:<port>/<jms_factory_name>/<jms_queue_name>?path=<ws_path>

Where "jms_factory_name" is a JMS factory, "jms_queue_name" is a JMS queue, and "ws_path" is the path to a deployed Web service.

From such a URL the JMS client transport knows which JMS factory and queue to use for the transmission of SOAP messages, and the JMS server transport knows the path, and hence the deployed service endpoint, to which inbound SOAP messages should be dispatched to. The "ws_path" is sent as a JMS message header with a request.

Currently Web services using the JMS transport component are deployed as standalone services and within the Glassfish application service for reasons of simplicity. To integrate with the application server requires that the standlone functionality is integrated into a message-driven bean that is responsible dispatching inbound messages to the deployed Web services.
