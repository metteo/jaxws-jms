Welcome to the JMS Web service transport project, an example Web service transport component for the JAX-WS reference implementation 2.0.1 that utilizes JMS.

Introduction

This project is designed to show by example how to implement plugable transports into the latest (rearchitectured) JAX-WS code, which supports client and service-based transports as plugable components.

The example chosen in this case is JMS. SOAP messages produced and consumed by the JAX-WS runtime are transmitted over a JMS Queues.

How to run the sample

A sample Web service is provided (see here XXXX) that is configured to use the JMS transport. The sample can be built and run in the following steps:

  • Build the JMS transport. Type 'ant' from the main project directory.

  • Create a JMS factory nameed 'jaxwsfactory' and a JMS queue named 'jaxwsqueue'. TODO instructions.

  • Run the standalone JMS container for hosting the Web service. Type 'ant start-server'.

  • Build and deploy sample Web service. Type 'ant clean server' from the samples/fromwsdl directory.

  • Build and run the client. Type 'ant clean client run-client' from the samples/fromwsdl directory.

How the sample works with the JMS transport

To "plug-in" a transport requires no modification to the following: client or service endpoint implementation code; and deployment descriptors. Everything is specified from the endpoint address URL, the schema of which determines what transport is used in JAX-WS.

The WSDL (AddNumbers.wsdl) used by the sample contains the following soap:address for the AddNumbersPort of the AddNumbersService: 

<soap:address location="jms://localhost:3700/jaxwsfactory/jaxwsqueue?path=fromjava/addnumbers" />

The location attribute value is a URL whose schema is "jms". This informs the JAX-WS client runtime that JMS client transport should be used. TODO on registration

The URL for the JMS transport has the following structure:

jms://<host>:<port>/<jms_factory_name>/<jms_queue_name>?path=<ws_path>

Where "jms_factory_name" is a JMS factory, "jms_queue_name" is a JMS queue, and "ws_path" is the path to a deployed Web service.

From such a URL the JMS client transport knows which JMS factory and queue to use for the transmission of SOAP messages, and the JMS server transport knows the path, and hence the deployed service endpoint, to which inbound SOAP messages should be dispatched to.
